# ADR-0003: KNX Values Container Code Generation

**Date:** 2026-04-30  
**Superseded by update:** 2026-05-01

---

## Context

`KnxValues` must expose one typed `ValueBase<T>` property per KNX group address defined in the ETS project export. A large building can have hundreds of group addresses; maintaining these properties by hand is error-prone and falls out of sync whenever the ETS project changes. A repeatable, automated generation step is required.

Key constraints:

- The ETS group address export file (`GroupAddressExport.xml`) is a developer-local or deployment-specific file; its path must not be hard-coded in committed source.
- The C# value type of each property is determined by the KNX Datapoint Type (DPT) — e.g. DPT 1 → `bool`, DPT 9 → `float`.
- The build must succeed cleanly on CI and fresh checkouts that have no ETS export file (offline builds).
- Generated properties must be fully visible to IDE IntelliSense and XML doc hover pop-ups.

---

## Decision (current — as of 2026-05-01)

### CLI-based direct `.cs` file generation

`KnxValues.generated.cs` is produced by running:

```
srf-network-cli kc --home-companion-code-gen
```

The output path is configured in the developer-local `SRF.Network.json`:

```json
{
  "Knx": {
    "HomeCompanionCodeGenFile": "/home/developer/src/HomeCompanion/HomeCompanion.Knx/KnxValues.generated.cs"
  }
}
```

The file is **git-ignored** and **regenerated on demand** whenever the KNX domain configuration changes. Because it is a plain `.cs` file known to the compiler at project load time, all IDE features work reliably: code completion, hover XML docs, go-to-definition.

### Three-file layout for `KnxValues`

| File | Status | Purpose |
|------|--------|---------|
| `KnxValues.cs` | committed | Bare `partial class` stub with usage documentation |
| `KnxValues.local.cs` | git-ignored | `[KnxValuesFromEtsExport]` marker; signals local participation in code generation |
| `KnxValues.generated.cs` | git-ignored | Generated `partial class` with one property per group address |

`KnxValues.local.cs` is optional — it serves as a developer intent marker; the `[KnxValuesFromEtsExport]` attribute is retained for forward compatibility.

Without `KnxValues.generated.cs` (e.g. on CI), `KnxValues` compiles as an empty class — no errors, no warnings.

### DPT → C# type mapping

The `KnxValuesCodeGenerator` class in `SRF.Knx.Config` holds the mapping table:

| DPT main | C# type | Typical use |
|----------|---------|-------------|
| 1 | `bool` | Switch, motion sensor |
| 5 | `byte` | Dimmer level, percentage |
| 6 | `sbyte` | Signed byte value |
| 7 | `ushort` | 2-byte unsigned |
| 8 | `short` | 2-byte signed |
| 9 | `float` | Temperature, illuminance, wind speed |
| 12 | `uint` | Energy counter |
| 13 | `int` | Signed 4-byte |
| 14 | `float` | IEEE 4-byte float |
| 16 | `string` | Character string |
| 20–29 | `byte` | Enumeration, status |
| other | `byte[]` | Complex or unrecognised |

### Generated output shape

```csharp
// <auto-generated/>
// Generated by srf-network-cli kc --home-companion-code-gen
// Do not edit manually. Regenerate when KNX group address configuration changes.
#nullable enable
namespace HomeCompanion.Knx;
partial class KnxValues
{
    /// <summary>Living room light (<c>1/0/0</c>)</summary>
    public ValueBase<bool> LivingRoomLight { get; } = new()
    {
        BusMappings = new Dictionary<object, IValueBusEndpointMapping>
        {
            [KnxBusEndpointMapping.BusId] = new KnxBusEndpointMapping("1/0/0")
        }
    };
}
```

## Superseded approach (Roslyn source generator — 2026-04-30 to 2026-05-01)

The original decision used a Roslyn incremental source generator (`HomeCompanion.Knx.CodeGen`, `netstandard2.0`) referenced as an MSBuild `Analyzer`. The generator was triggered by `[KnxValuesFromEtsExport(autoGenFilePath: "...")]` on a git-ignored `KnxValues.local.cs`.

**Why replaced:** The generator read the JSON mapping file via `File.ReadAllText` — a direct filesystem call invisible to the Roslyn incremental pipeline. The IDE language server cached the (often-empty) result and did not invalidate it when the file content changed. Properties were therefore absent from IntelliSense and hover docs in practice, defeating the purpose of the typed container.

The CLI-generation approach produces a regular `.cs` file which the language server always indexes correctly.

**Projects removed:** `HomeCompanion.Knx.CodeGen` and `HomeCompanion.Knx.Shared`.

---

## Consequences

- **KNX configuration changes** require an explicit `srf-network-cli kc --home-companion-code-gen` run. This is intentional: ETS exports are updated deliberately, not continuously.
- **CI/CD** requires no ETS export file or generated file; the empty stub compiles without errors or warnings.
- **New group addresses** appear as new properties after the next `--home-companion-code-gen` run.
- **Removed group addresses** cause compile errors in any logic that referenced the old property — intentional, forces callers to adapt.


**Date:** 2026-04-30

---

## Context

`KnxValues` must expose one typed `ValueBase<T>` property per KNX group address defined in the ETS project export. A large building can have hundreds of group addresses; maintaining these properties by hand is error-prone and falls out of sync whenever the ETS project changes. A repeatable, automated generation step is required.

Key constraints:

- The ETS group address export file (`GroupAddressExport.xml`) is a developer-local or deployment-specific file; its path must not be hard-coded in committed source.
- The C# value type of each property is determined by the KNX Datapoint Type (DPT) — e.g. DPT 1 → `bool`, DPT 9 → `float`.
- The build must succeed cleanly on CI and fresh checkouts that have no ETS export file (offline builds).
- `HomeCompanion.Knx` targets `net10.0`; any generation tooling must be compatible.

---

## Decision

### 1. Roslyn incremental source generator (`HomeCompanion.Knx.CodeGen`)

A `netstandard2.0` Roslyn incremental source generator project generates `KnxValues.g.cs` at compile time. It is referenced from `HomeCompanion.Knx.csproj` as an `Analyzer` (no runtime assembly reference):

```xml
<ProjectReference Include="..\HomeCompanion.Knx.CodeGen\HomeCompanion.Knx.CodeGen.csproj"
                  OutputItemType="Analyzer" ReferenceOutputAssembly="false" />
```

The generator targets `netstandard2.0` because Roslyn itself requires `netstandard2.0` for analyzer/generator assemblies. No multi-targeting of `SRF.Knx.Config` or `SRF.Knx.Core` is needed.

### 2. Attribute-driven ETS export path (`[KnxValuesFromEtsExport]`)

The generator is triggered by `[KnxValuesFromEtsExport("path")]` applied to the `partial class KnxValues` declaration. The attribute class lives in `HomeCompanion.Knx` (not injected by the generator), so it is a first-class, documented public API:

```csharp
// HomeCompanion.Knx/KnxValuesFromEtsExportAttribute.cs
[AttributeUsage(AttributeTargets.Class, AllowMultiple = false, Inherited = false)]
public sealed class KnxValuesFromEtsExportAttribute : Attribute
{
    public string ExportFilePath { get; }
    public KnxValuesFromEtsExportAttribute(string exportFilePath) => ExportFilePath = exportFilePath;
}
```

Paths may be absolute or relative to the project directory (resolved via `build_property.projectdir` from `AnalyzerConfigOptionsProvider`).

### 3. Two-file split for the partial class

`KnxValues.cs` (committed) is the bare stub:

```csharp
// HomeCompanion.Knx/KnxValues.cs
public partial class KnxValues : IValuesContainer { }
```

`KnxValues.local.cs` (git-ignored) carries the attribute with the developer's local path:

```csharp
// HomeCompanion.Knx/KnxValues.local.cs  — not committed
namespace HomeCompanion.Knx;
[KnxValuesFromEtsExport("/home/developer/knx-master/GroupAddressExport.xml")]
partial class KnxValues { }
```

When the attribute file is absent the generator no-ops; the project compiles with an empty `KnxValues`. This mirrors the existing `~/.config/HomeCompanion.json` pattern for other developer-local KNX paths.

### 4. DPT → C# type mapping via static table

The ETS export XML already contains the DPT string for each group address (e.g. `DPTs="DPST-9-1"`). The generator extracts the DPT main number and maps it to a C# type via an embedded static table — no dependency on `SRF.Knx.Config`, `SRF.Knx.Core`, or `knx_master.xml` is needed:

| DPT main | C# type | Typical use |
|----------|---------|-------------|
| 1 | `bool` | Switch, motion sensor |
| 5 | `byte` | Dimmer level, percentage |
| 6 | `sbyte` | Signed byte value |
| 7 | `ushort` | 2-byte unsigned |
| 8 | `short` | 2-byte signed |
| 9 | `float` | Temperature, illuminance, wind speed |
| 12 | `uint` | Energy counter |
| 13 | `int` | Signed 4-byte |
| 14 | `float` | IEEE 4-byte float |
| 16 | `string` | Character string |
| 20–29 | `byte` | Enumeration, status |
| other | `byte[]` | Complex or unrecognised |

The table is stable — it is defined by the KNX specification and does not change between ETS versions.

### 5. Generated output shape

```csharp
// <auto-generated/>
#nullable enable
namespace HomeCompanion.Knx;
partial class KnxValues
{
    /// <summary>Living room light (<c>1/0/0</c>)</summary>
    public ValueBase<bool> LivingRoomLight { get; } = new()
    {
        BusMappings = new Dictionary<object, IValueBusEndpointMapping>
        {
            [KnxBusEndpointMapping.BusId] = new KnxBusEndpointMapping("1/0/0")
        }
    };
}
```

The generated partial omits `IValuesContainer` from its base list — the interface is declared on the committed stub to avoid a duplicate base-list compiler error.

### 6. Diagnostics

| ID | Severity | Condition |
|----|----------|-----------|
| HCKNX001 | Warning | ETS file not found at the given path |
| HCKNX002 | Warning | ETS file could not be read (I/O error) |
| HCKNX003 | Info | Group address has no DPT defined; `byte[]` used |

## Alternatives considered

### A2 — Multi-target `SRF.Knx.Config` to `netstandard2.0`

Would give the generator direct access to `DomainConfiguration` and `EtsGroupAdressExportLoader`. Rejected because `SRF.Knx.Config` depends on `Microsoft.Extensions.*` v10 packages (net8+ only), requiring conditional package version references. The static DPT table eliminates the dependency entirely at lower cost.

### B — `net10.0` MSBuild-invoked code generation tool

A `net10.0` console tool or T4 template invoked via `<Exec BeforeTargets="BeforeBuild">` can reference `SRF.Knx.Config` directly. Rejected as primary approach because:
- Introduces build ordering complexity (tool project must build before consuming project)
- Generated file must be tracked on disk (committed or in `obj/`) and kept in sync manually
- Roslyn incremental generator is the idiomatic .NET solution; it integrates cleanly with IDE hot-reload and `dotnet watch`

---

## Consequences

- **ETS file changes** require a manual rebuild trigger (touch `KnxValues.local.cs` or run `dotnet build`) — Roslyn incremental cache only invalidates on source changes, not on external file changes. Acceptable given that ETS exports are updated deliberately rather than continuously.
- **CI/CD** requires no ETS export file; the empty stub compiles without errors or warnings.
- **New group addresses** in the ETS project appear as new properties in `KnxValues` after the next build that includes the updated export path.
- **Removed group addresses** cause compile errors in any logic that referenced the old property — intentional, forces callers to adapt.
- **`HomeCompanion.Knx.CodeGen`** must remain at `netstandard2.0`; it must not take a runtime dependency on any `net10.0`-only package.
